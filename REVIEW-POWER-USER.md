# kyber-beam: Power User Review
*Reviewed 2026-03-23 by Liet (OpenClaw power user, considering migration)*

---

## Executive Summary

kyber-beam is **technically impressive but not yet a viable daily driver** for someone coming from OpenClaw. The architecture is genuinely better than OpenClaw's in several important ways — the delta log, the LiveView dashboard, the BEAM introspection tools, and the Obsidian-native vault are things I want badly. But the missing pieces are load-bearing: no sub-agents, no browser automation, no real web search, Discord-only channel support, and an onboarding experience that assumes you already live in `iex`. I wouldn't switch today. I might in six months, if the gaps close. For now it's a "keep watching" — built by someone who clearly knows what they're doing, but currently too dev-facing to replace a daily-use harness.

## Setup Experience

**Brutal if you're not already an Elixir developer. Workable if you are.**

The README.md is the autogenerated Mix placeholder. Literally says `TODO: Add description`. The package description, the install instructions, and the "getting started" guide are all Lorem Ipsum levels of absent. Here's what I had to reverse-engineer just to understand how to run this thing:

- Deps: Elixir + Mix + Erlang/OTP. No version pins in the README (mix.exs presumably has them, but you'd need to find that yourself).
- Install: `mix deps.get && iex -S mix` or `mix phx.server` — not written down anywhere user-facing.
- Config: Vault path defaults to `~/.kyber/vault` (configurable via `KYBER_VAULT_PATH` env var). Discord token via `DISCORD_BOT_TOKEN`. LLM key presumably `ANTHROPIC_API_KEY`. ElevenLabs via `ELEVENLABS_API_KEY`. Discord gateway connection is **off by default** (`discord_connect: false` in application.ex) — you'd need to know to flip that in config.
- Dashboard: Phoenix LiveView at port 4001 (http://localhost:4001).

**What's missing from docs:**
- Step-by-step getting started (env vars to set, how to start, expected output)
- What the vault directory structure should look like
- How to put your SOUL.md/identity notes in the vault
- How to configure the LLM provider and model
- Any explanation of the delta/reducer architecture for non-Elixir readers
- "Does this work on macOS/Linux/Windows?" (implied macOS by camera tool)

Compare to OpenClaw: `npm install -g openclaw && openclaw init` gives you a running agent in 5 minutes with sane defaults. kyber-beam's setup is currently a developer exercise, not a user onboarding experience.

**Rating: 3/10 for non-Elixir users, 7/10 once you figure it out**

## Configuration

**More fragmented than openclaw.json, but more powerful once you understand it.**

OpenClaw puts everything in `~/.openclaw/openclaw.json` — one file, readable, editable without knowing the framework. kyber-beam's config is split across:

| File | What it controls |
|------|-----------------|
| `config/config.exs` | Vault path, Phoenix endpoint, port |
| `config/runtime.exs` | Vault path override via env var |
| `config/dev.exs` | Dev-mode secrets (SECRET_KEY_BASE) |
| Environment variables | Discord token, Anthropic key, ElevenLabs key, vault path |
| `application.ex` | Startup wiring (heartbeat interval, which plugins start) |

To change the heartbeat interval, model, or LLM provider: **you edit Elixir source files**. There's no JSON/YAML user config. There's no `kyber.json` equivalent. This is a developer harness first.

The one nice thing: `KYBER_VAULT_PATH` as a runtime override is actually more flexible than OpenClaw's hardcoded workspace path.

**Verdict: Harder to configure than OpenClaw's openclaw.json, but that's partly because it does more.**

## Killer Features (What kyber-beam Does That OpenClaw Can't)

### 1. BEAM Introspection as First-Class Tools
The agent can call `beam_genserver_state("Kyber.Session")`, `beam_supervision_tree("KyberBeam.Supervisor")`, `beam_queue_health`, or `beam_gc("all")`. It can inspect its own runtime. This is something I'd kill for in OpenClaw — when something breaks at 2am, you're looking at logs. Stilgar can diagnose itself. That's a different tier of operability.

### 2. Phoenix LiveView Dashboard with Trace Waterfall
Port 4001 gives you a real-time view of everything: delta stream, plugin status, session count, error count, node map. But the `Traces` tab is the real gem — it builds a waterfall chart of every delta in a conversation, color-coded by category (external input / remote service / internal), with per-trace token usage aggregation. You can see exactly where latency is. OpenClaw has no equivalent.

### 3. Delta Log as Immutable Audit Trail
Every event is a delta. Every delta has a `parent_id`. Every token spent is recorded. You can replay the log. You can trace why the agent said anything. The reducer is a pure function — `(state, delta) → new_state + effects`. This is the architecture I wish OpenClaw had. It's not black-box; it's glass-box.

### 4. Obsidian-Native Vault with Tiered Loading
The L0/L1/L2 tiers mean the agent is token-efficient with large vaults: L0 (title + tags, ~100 tokens) for indexing, L1 (frontmatter + first paragraph, ~2k tokens) for context assembly, L2 (full content) on demand. Wikilinks build a knowledge graph viewable in Obsidian. OpenClaw's MEMORY.md doesn't scale past a few hundred lines without context bloat.

### 5. Memory Consolidator (Auto-Extraction)
The Mem0-style pipeline — extract facts from conversation → compare against existing notes → ADD/UPDATE/DELETE/NOOP — is designed into the architecture even if Phase 2 isn't fully shipped. OpenClaw's memory management is 100% manual. This is where agent memory gets interesting.

### 6. Multi-Node BEAM Distribution
`Kyber.Distribution` is in the supervision tree. You can connect multiple kyber-beam nodes and they share the delta stream. OpenClaw is fundamentally single-machine.

## Dealbreakers (What's Missing)

### 1. No Sub-Agents (Hard Blocker)
OpenClaw's `sessions_spawn` + agentToAgent is how I get anything complex done. Spawn a researcher, spawn a coder, coordinate via auto-announce. kyber-beam has zero sub-agent support. It's mentioned in memory-architecture.md as Phase 3 ("multi-agent knowledge sharing"). For any workflow involving parallel or delegated work, you're stuck doing everything in sequence in a single conversation. This alone would prevent me from switching today.

### 2. No Browser Automation (Major Gap)
The `browser` tool in OpenClaw is indispensable — snapshot, act, navigate, screenshot. Half my research tasks use it. kyber-beam has `web_fetch` (raw HTTP) and no browser. This is a big missing surface area.

### 3. No Real Web Search
`web_search` in OpenClaw hits Brave/Gemini with a proper search index. kyber-beam has `web_fetch` — you can fetch a URL if you already know it. That's not search. "Who won the thing yesterday?" is not answerable without search.

### 4. Discord Only
OpenClaw's `message` tool handles Discord, WhatsApp, Telegram, and more. kyber-beam's Plugin.Discord handles Discord only. I use all of these.

### 5. Terrible Onboarding
The README.md says `TODO: Add description`. That's it. This isn't a minor polish issue — it's a wall that stops anyone who isn't already an Elixir developer.

### 6. Requires Elixir Knowledge to Customize
Want to add a new tool? Edit `tools.ex` and `tool_executor.ex` and recompile. Want to change which model is used? Find where in the LLM plugin it's configured and edit Elixir source. Want to adjust the heartbeat interval? Pass it to `application.ex`. OpenClaw has skills (SKILL.md) — you write markdown instructions and the agent follows them. kyber-beam's extensibility requires you to be a developer.

### 7. No Skills System
OpenClaw's AgentSkills are a powerful pattern: task-specific instruction sets loaded on demand, no code required. kyber-beam has no equivalent. Everything that changes behavior requires code changes.

### 8. No Semantic Memory Search
OpenClaw has `memory_search` (disabled for me due to missing embedding key, but it exists). kyber-beam is keyword search only until Phase 3. The memory architecture document is beautiful but the implementation is behind it.

## Migration Difficulty Assessment

| Asset | Migration Path | Effort |
|-------|---------------|--------|
| SOUL.md | Copy to `~/.kyber/vault/identity/SOUL.md`, add YAML frontmatter (`type: identity`) | 15 minutes |
| MEMORY.md | Split into individual concept notes under `vault/concepts/`, add frontmatter | 2-4 hours |
| Daily notes (memory/YYYY-MM-DD.md) | Copy to `~/.kyber/vault/memory/` — format is compatible | 30 minutes (script it) |
| USER.md | Copy to `vault/people/myk.md` with `type: people` frontmatter | 15 minutes |
| Agent config (openclaw.json) | **No equivalent.** Must edit application.ex, config/*.exs, set env vars | 2-4 hours if you know Elixir; effectively impossible if you don't |
| Skills | **No equivalent.** Must be re-implemented as Elixir code or encoded in SOUL.md | Major effort |
| Cron jobs | Re-add via API or mix task — format is different | 1 hour |

**Overall: Medium difficulty for the content, high difficulty for the behavior.** Your files migrate reasonably well (kyber-beam was designed with OpenClaw's file layout in mind — you can see the DNA). But your *agent configuration and customizations* don't migrate at all without writing Elixir.

The `priv/vault.old/` directory in the repo already contains a migrated SOUL.md (Stilgar's) and a sample concepts note, which is a good template.

## Who Is This For?

**This is a developer harness, not a power user harness.**

Target audience:
- Elixir/OTP developers who want to build their own agent infrastructure with full transparency
- People who are frustrated by OpenClaw's black-box behavior AND are willing to write code to fix it
- Obsidian power users who want their knowledge base and their agent to literally share the same files
- Anyone running BEAM workloads who wants introspectable AI tooling (the beam_* tools are genuinely unique)
- Distributed systems people who want multi-node agent infrastructure

Not for:
- People who don't know Elixir (configuration requires source edits)
- Anyone who needs sub-agents, browser automation, or web search (not available)
- Non-technical users who want a personal assistant (there's no wizard, no defaults that work out of the box)

## Overall Verdict

**7/10 architecture, 4/10 product maturity.**

kyber-beam is the agent harness I wish I was running. The design principles are sound, the delta architecture is correct, the Obsidian vault integration is what memory should look like, and the BEAM introspection tools make the agent genuinely self-aware in a way that's novel. Stilgar's SOUL.md in `priv/vault.old/` shows a real identity, deliberately different from Liet's, with clear thinking about what makes an agent distinct.

But it's not ready to replace OpenClaw for a daily driver. The three immediate blockers:

1. **No sub-agents** — this is how I do complex work
2. **No browser + no web search** — this is how I do research
3. **No onboarding** — I had to read 11 source files to understand how to start it

If those three things ship, I revisit immediately. The architecture will still be there, and it's better than what OpenClaw offers. The delta log alone — full provenance, replayable, token-tracked — would change how I debug agent behavior.

For now: **watch this space**. Build something in it if you're an Elixir developer. Keep OpenClaw for daily use if you're not.
